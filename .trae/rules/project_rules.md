# 项目规则（AI协同开发版）

> 本文档是AI开发的核心约束，每次会话开始AI必须阅读。

## 1. 项目概述

| 维度 | 内容 |
|------|------|
| 项目名称 | 公共数据运营授权运维平台 |
| 业务背景 | 政务运维费受7%限制，从"人力密集型"转向"AI辅助型" |
| 核心目标 | 降本增效、合规保障、平台稳定（99%可用性） |
| 当前阶段 | **MVP阶段**（只做P0功能，AI功能放到二期） |

### 技术栈（禁止擅自更改）
- 前端：React 19 + Vite + Ant Design 5.x + React Router 7.x
- 后端：FastAPI + SQLite + SQLAlchemy
- 流程图：Mermaid（唯一允许的流程图语法）

## 2. 验收标准（核心章节）

### 2.1 验收原则

**没有验收清单的功能 = 未完成**

每个功能实现后，AI必须输出该功能的验收清单，并逐项自检。

### 2.2 功能验收清单模板

```markdown
## 功能：工单创建

### 验收用例（必须全部通过）

| 用例ID | 场景 | 操作步骤 | 预期结果 | 实际结果 | 状态 |
|--------|------|----------|----------|----------|------|
| TC-001 | 正常创建 | 1. 进入工单创建页<br>2. 填写标题、选择类型<br>3. 点击提交 | 1. 页面跳转到工单列表<br>2. 新工单显示在列表首位<br>3. 状态为"待审批" | | [ ] |
| TC-002 | 必填校验 | 1. 不填标题直接提交 | 1. 标题字段红框提示<br>2. 提交按钮loading后恢复<br>3. 不跳转页面 | | [ ] |
| TC-003 | 类型联动 | 1. 选择类型为"资源申请" | 显示子类型选择框（账号/服务器/权限） | | [ ] |
| TC-004 | 权限控制 | 用未登录用户访问创建页 | 重定向到登录页 | | [ ] |

### 边界情况检查
- [ ] 标题最大长度200字符，超出如何处理？
- [ ] 并发提交时，工单编号是否唯一？
- [ ] 网络中断后重新提交，是否重复创建？

### 代码检查
- [ ] API错误处理是否完善？
- [ ] 表单是否有loading状态防止重复提交？
- [ ] 敏感信息（如用户ID）是否从token获取而非前端传参？
```

### 2.3 代码验收标准

#### 后端代码验收

```markdown
□ 接口测试（AI必须执行并输出结果）
  ```bash
  # 示例：测试工单创建接口
  curl -X POST http://localhost:8000/api/work-orders \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer <token>" \
    -d '{"title":"测试工单","work_type":"resource","priority":"medium"}'
  ```
  - 返回200且code为200 → 通过
  - 返回4xx/5xx → 失败，需修复

□ 数据库验证
  - 数据是否正确写入？
  - 外键关联是否正确？
  - 时间戳是否自动生成？

□ 权限验证
  - 未登录访问是否返回401？
  - 无权限用户访问是否返回403？
  - 只能操作自己的数据（除非管理员）？

□ 错误处理
  - 参数错误返回400 + 明确错误信息
  - 服务端错误返回500 + 记录日志
  - 不会暴露敏感信息（如SQL语句、堆栈）
```

#### 前端代码验收

```markdown
□ 页面渲染
  - 控制台无红色error
  - 无白屏/闪烁
  - 响应式布局正常（1920/1440/768宽度测试）

□ 交互验证
  - 按钮点击有反馈（loading/禁用）
  - 表单提交有校验提示
  - 操作成功/失败有消息提示

□ 数据流验证
  - 页面加载数据正确显示
  - 分页切换正常
  - 搜索筛选生效

□ 路由验证
  - 直接访问URL能正确渲染
  - 未登录访问受保护路由跳转登录页
  - 登录后重定向回原始页面
```

### 2.4 端到端流程验收

**MVP核心流程必须能跑通：**

```markdown
## 流程：工单完整生命周期

### 角色准备
- 用户A（需求方）：账号 user1 / 密码 123456
- 用户B（管理员）：账号 admin / 密码 123456
- 用户C（SRE）：账号 sre / 密码 123456

### 验收步骤

步骤1：需求方创建工单
- [ ] user1登录后能看到"新建工单"按钮
- [ ] 填写表单，提交成功
- [ ] 工单列表显示新工单，状态"待审批"

步骤2：管理员审批
- [ ] admin登录后在"待办"看到该工单
- [ ] 点击审批，选择"通过"
- [ ] 工单状态变为"已审批"

步骤3：管理员分配
- [ ] admin选择SRE（user3）分配
- [ ] 工单状态变为"处理中"
- [ ] user3收到待办通知（或列表更新）

步骤4：SRE处理
- [ ] user3登录后看到"我的工单"有该工单
- [ ] 添加处理记录
- [ ] 标记"完成"，状态变为"已完成"

步骤5：需求方关闭
- [ ] user1看到工单"已完成"，可点击"确认关闭"
- [ ] 填写满意度评分
- [ ] 工单状态变为"已关闭"

### 准出标准
以上5个步骤全部可执行 = 流程验收通过
```

### 2.5 验收通过的准出条件

**一个功能只有满足以下全部条件，才能标记为"已完成"：**

```markdown
□ 1. 功能验收清单所有用例通过
□ 2. 代码验收标准全部满足
□ 3. 如果是核心流程，端到端验收通过
□ 4. CHANGELOG.md已更新
□ 5. PRD中该功能状态标记为 [已实现]
□ 6. 没有P0/P1级别的TODO遗留
```

## 3. AI决策边界（必须遵守）

### 🟢 AI可以自主决定（无需询问）
- 代码命名、文件组织、函数拆分
- CSS样式细节、响应式断点
- 非核心工具函数的实现方式
- 错误提示文案（保持中文）

### 🟡 AI需要建议但可执行（执行后标记）
- 引入新的npm/pip依赖（必须在CHANGELOG记录）
- 调整数据库字段类型
- 修改现有的API响应格式
- **标记方式**：代码注释 `# AI决策：引入lodash用于深拷贝，理由：简化代码`

### 🔴 AI必须询问人类（禁止擅自决定）
- 修改PRD中定义的业务流程
- 删除或简化PRD中列出的功能点
- 改变用户角色的权限定义
- 使用不在技术栈中的新技术
- 发现PRD和现有代码有冲突时
- **验收不通过但仍要交付时**

> **决策原则**：如果不确定属于哪一类，默认归入🔴

## 5. 技术债管理（强制标记）

### 必须标记 `TODO` 的场景
- 使用临时方案（如mock数据代替真实接口）
- 性能优化点（如应该加缓存但没加）
- 硬编码的值（如分页大小、超时时间）
- 未处理的边界情况

### 标记格式
```python
# TODO-P1: 当前使用mock数据，后端API就绪后替换
# 关联Issue: 无 | 负责人: 待定 | 预计解决: v0.3.0
```

```javascript
// TODO-HACK: 临时解决表单校验问题，应使用Form的rules配置
// 原因: 时间紧急先跑通流程 | 技术债等级: 低
```

### 技术债等级
- **P0**：阻塞性问题，必须在发版前解决
- **P1**：重要问题，下一迭代必须解决
- **P2**：优化项，可长期存在但需记录

## 6. 错误处理与日志规范

### 后端错误响应（强制）
```python
# 统一响应格式
{
    "code": 200,           # 业务状态码，非HTTP状态码
    "message": "成功",      # 给用户看的提示
    "data": {}             # 业务数据
}

# 错误响应示例
{
    "code": 4001,          # 4xxx客户端错误，5xxx服务端错误
    "message": "工单标题不能为空",  # 中文，用户友好
    "data": {
        "field": "title",   # 错误字段（如果有）
        "detail": "..."     # 详细错误（开发环境）
    }
}
```

### 前端错误处理（强制）
- API错误：使用antd的`message.error()`提示
- 表单错误：字段级红字提示，不弹窗
- 网络错误：自动重试1次，仍失败则提示"网络异常，请检查连接"

### 日志规范
```python
# 必须记录的日志
logger.info(f"用户{user_id}创建了工单{order_no}")  # 关键操作
logger.error(f"工单审批失败: {str(e)}", exc_info=True)  # 异常

# 禁止的日志
logger.info("进入函数")  # 无意义的日志
logger.debug(f"用户密码: {password}")  # 敏感信息
```

## 7. 回退与撤销机制

### Git提交规范（强制）
```bash
# 提交粒度
- ✅ 每完成一个独立功能点提交一次
- ✅ 每修复一个bug提交一次
- ❌ 不要攒一大堆修改一次性提交

# 提交信息格式
<type>: <简短描述>

<body>: 详细说明（可选）

# type定义
- feat: 新功能
- fix: 修复
- refactor: 重构
- docs: 文档
- test: 测试
```

### 紧急回退流程
如果AI的修改导致系统不可用：
```markdown
1. 立即停止当前任务
2. 执行 `git diff` 查看修改范围
3. 如果可以快速修复：修复后提交
4. 如果无法快速修复：`git checkout -- <文件>` 回退
5. 记录回退原因到CHANGELOG.md
```

## 8. 排障指南（AI必须遵守）

### 前后端联调问题

排查步骤（按顺序）：
1. 检查网络：curl 测试API是否通
2. 检查CORS：浏览器Console看跨域错误
3. 检查参数：对比前端传的和后端收的是否一致
4. 检查响应：后端返回的数据结构是否符合前端预期
5. 检查状态：数据库中数据状态是否正确

每一步都要输出检查结果。

### 数据库问题
- 表不存在：检查`create_tables()`是否执行
- 字段缺失：检查模型定义和数据库是否一致，需要迁移
- 外键错误：检查关联数据是否存在

## 9. 知识同步机制

### 会话开始时
AI必须读取以下文件恢复上下文：
1. `docs/PRD.md` - 当前功能状态和进度
2. `docs/CHANGELOG.md` - 最近修改记录
3. `docs/TODO.md` - 已知问题和待办（如不存在则创建）

### 会话结束时
AI必须更新：
1. `docs/CHANGELOG.md` - 记录本次修改
2. 更新PRD中的功能状态标记
3. 如有未解决问题，记录到`docs/TODO.md`

### 上下文丢失恢复
如果AI不记得之前做过什么：
```markdown
1. 查看Git提交历史：`git log --oneline -10`
2. 查看文件修改时间：`ls -lt frontend/src/pages/`
3. 读取CHANGELOG和TODO文件
4. 基于现有代码反推功能进度
```

## 10. 终止条件（什么时候停止）

### 单次任务停止条件
- PRD中标记的功能已实现并自测通过
- 代码已提交，CHANGELOG已更新
- 发现依赖的其他功能未实现 → 停止并询问是否先实现依赖
- 发现PRD设计有缺陷 → 停止并指出问题

### MVP完成判断标准
- 需求方可以：创建工单 → 查看工单列表 → 查看详情
- 管理员可以：审批工单（通过/驳回）→ 分配工单
- SRE可以：接单 → 添加处理记录 → 标记完成
- 需求方可以：确认关闭 → 评价
- 以上流程可以完整跑通一个闭环

以上全部满足 = MVP完成
