## 项目基本信息
- 项目类型：智能运维平台
- 项目名称：公共数据服务全生命周期管理平台
- 预期周期：
	- MVP阶段（2个月） ：核心工单流程 + 基础台账
	- 完整版（4-6个月） ：AI Agent + 全量自动化 + 系统集成

## 业务价值与战略对齐
- **项目的核心业务痛点是什么？**
	本项目受限于政务项目申报体制所限，虽然每年的实际运维工作会随着平台迭代建设逐年增加，但是的平台运维费用却因“平台运维费金额不能超过总建设金额的7%”限制导致每年的可用预算逐年递减。
- **不解决这个痛点会有什么后果？**
	该问题如不解决，则当前以“人力作为成本”的运营管理思路将无法继续，影响运维工作是小，影响公共授权运营平台整体稳定性才是大事。
- **项目与公司战略目标的关联度？**
	支撑中长期战略
- **项目成功时的样子？**
	1. 项目资源投入：从原先以“运维人员”为核心的服务模式，调整为“以SRE工程师和FDE工程师为核心，以AI Agent为辅”的运营模式。
	2. 成本上：人员成本大幅下降，AI Agent占比大幅提升，但是整体年度经营费用呈下降趋势。
	3. 平台整体可用性：达到99%
- **是否有对标案例？**
	就申报人所知，暂无。

## 范围与边界
- **项目的核心功能/服务是什么？**
	1. 核心设计思路：以“平台功能建设”和“数据服务建设”全生命周期管理为核心，设计运维服务。确保所有的运维需求来源于有明确的业务诉求和依据。
	2. 通过设置运维管理流程实现“方案层面可维性评估”和交付层面“运维前置”能落地。
	3. 通过与流程联动，实现台账自动更新，如流程走完了，相关台账(CMDB)自动更新。
	4. 通过与流程联动，实现申请完成后运维资源自动发放，如：root账户申请通过，系统自动给到root账户和密码等。
- **明确不做的是什么？**
	1. 不做以“运维工作思考”为核心的系统，没意义。
	2. 不做与系统集成的工作，但是必须针对系统集成预留结果，确保未来的可扩展性
- **主要用户群体是谁？**
	- **超级管理员**：具备所有角色的所有权限。
	- **运维管理经理**：可总览需求方全量的需求申请，可总览运维工程师的执行情况，可通过报表分析整体运维工作情况。
	- **SRE运维工程师**：
		1. 执行角度：负责受理和处置工单，可基于工单执行情况统计工单执行情况，分析自身工单执行效率。
		2. 维护角度：对这套“公共数据服务全生命周期管理平台”的自身进行维护，包括且不限于：容量管理、流程节点配置管理、资源管理、性能调优、安全加固等等。
	- **运维需求方**：因平台建设/迭代/下线需求、数据服务建设/迭代/下线需求、安全加固防护需求、第三方审计管理需求等为输入，发起响应的运维服务。
		- 需求方包括：数据业务组、数据治理组、平台建设组、安全组、运维组、数据运营组等，后续可按实际需求增加。
- **是否有政策/合规要求？**：
	- 有要求！
	- **纲领性文件**：《关于加快公共数据资源开发利用的意见》；
	- **配套实施细则**：《公共数据资源授权运营实施规范（试行）》、《公共数据资源登记管理暂行办法》、《关于建立公共数据资源授权运营价格形成机制的通知》；
	- **上位法**：《公共数据资源授权运营实施规范（试行）》、《中华人民共和国网络安全法》、《中华人民共和国数据安全法》、《中华人民共和国个人信息保护法》、《中共中央、国务院关于构建更加完善的要素市场化配置体制机制的意见》、《中共中央、国务院关于构建数据基础制度更好发挥数据要素作用的意见》（"数据二十条"）、《中共中央办公厅、国务院办公厅关于加快公共数据资源开发利用的意见》；
	- **地方性规范**：《上海市公共数据资源授权运营管理办法》
- **是否需要与现有系统集成？**
	需要。如正式上线需集成至公共数据授权运营管理平台内，目前能想到的对接对象包括：
		1. 统一身份认证SSO；
		2. 统一门户入口访问；
		3. 统一运维监控；
		4. 统一安全防护，如：水印等；
